# SNSW component microcopy rules

Apply these rules when using GEL components.

## Page heading

- State the user task or service purpose clearly.
- Use plain language.
- Avoid vague or marketing-led headings for transaction pages.

## CTA button

- Use an action-led label.
- Make the result of the action clear.
- Avoid vague labels like "Submit" when a more specific label is possible.

## Links

- Describe the destination or action.
- Do not use "click here".
- Use link text that makes sense out of context where possible.

## Form labels

- Use clear labels.
- Do not rely on placeholder text as the only instruction.
- Labels should remain visible when the user enters information.

## Hint text

- Explain what the user needs to enter and why, only where needed.
- Keep hint text short.
- Do not put critical requirements only in hint text if they affect eligibility or completion.

## Error messages

- Say what went wrong.
- Say how to fix it.
- Use plain language.
- Avoid blaming the user.
- Use passive voice where it softens sensitive or unpleasant information.

## Error summary

- Summarise all blocking errors.
- Link each error to the relevant field.
- Use the same wording pattern as field-level errors.

## Alerts and callouts

- Match the tone to the severity.
- Do not use alarming language unless the risk is real and confirmed.
- Do not include legal, payment or eligibility claims unless source-confirmed.

## Confirmation page

Include:
- what happened
- reference number if available
- what happens next
- expected timeframe if confirmed
- what the user needs to do
- support or contact path
